# Proposal — Per-news micro-pages for SEO

> Status: **draft proposal, not implemented**. Created for discussion.
> Related: SEO_RECOVERY.md, IndexNow setup (already shipped).

## Goal

Convert the news feed (currently rendered client-side from `/api_v2?action=news`) into a set of crawlable, indexable, **value-adding** landing pages — one per news item — that:

1. Rank in Google / Bing for the news headline + adjacent crypto keywords.
2. Funnel users from search into the GasNow dashboard.
3. Trigger IndexNow on every new item, so Bing / Yandex / Naver / Seznam reindex within minutes.
4. Are **not thin content** (otherwise Google penalizes — see "Why this is risky" below).

## Why this is risky

Naïvely shipping `/news/<slug>.html` with just `headline + link to source` will get the site classified as **doorway pages** / **thin content scraping**, which can tank ranking for the whole domain. We must add unique value per page. Bare minimum:

- Original short editorial framing (~80–150 words) tying the news to a current GasNow indicator (e.g., "BlackRock files spot ETH ETF — gas at 12 Gwei, MSI at 34/100 suggests calm conditions for ETF inflow scenarios").
- Live data widget snapshot at time of publication (gas prices, Fear & Greed, MSI).
- Outbound link to the original source (`rel="nofollow noopener"` so we don't bleed link equity).
- Two or three internal links to related GasNow widget anchors.
- `NewsArticle` Schema.org with `datePublished`, `headline`, `image`, `publisher` (GasNow).

If we can't ship the editorial layer, **abandon this proposal**. Better no micro-page than a thin-content one.

## Architecture options

### Option A — Static generation on a schedule (recommended)

1. **Cron worker** (Supabase Edge Function, GitHub Action, or simple node-cron container) runs every 15-30 minutes.
2. Fetches `/api_v2?action=news`. Deduplicates against a stored manifest.
3. For each new item:
   - Generates `dist/news/<source-slug>-<headline-slug>.html` from a template.
   - Editorial framing is generated by:
     - **(easy)** rule-based template: "{{headline}}. As of {{datetime}}, ETH gas was {{eth_gas}} Gwei and the Market Stress Index stood at {{msi}}/100, indicating {{msi_label}} market conditions."
     - **(better)** LLM-generated 80-word framing (Claude Haiku or local model), one-shot prompt with the headline + current indicators as input.
4. Updates `news-sitemap.xml` (Google News sitemap, separate from main sitemap).
5. Pings IndexNow with the URLs of new items only.

**Pros**: simple, no SSR runtime, nginx serves plain HTML. Cacheable infinitely. Crawler-perfect.
**Cons**: requires regular cron + a persistent dedup store (Supabase table or a JSON manifest in a volume).

### Option B — On-demand SSR (Cloudflare Worker or Edge Function)

Render `/news/<slug>` on the fly when a crawler hits it. Cache aggressively.

**Pros**: no cron, always fresh.
**Cons**: cold-start latency, needs a route layer in front of nginx, complicates the simple Docker deployment.

### Option C — Hybrid (React Router + prerender)

Add React Router to the SPA, create a `/news/:slug` route, run `vite-plugin-prerender` at build time against a list of known news items.

**Pros**: keeps SPA model.
**Cons**: build time grows linearly with news items; rebuild required for new items (defeats the freshness goal).

**Recommendation: Option A.**

## Required schema per page

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "NewsArticle",
  "headline": "...",
  "datePublished": "2026-06-03T14:32:00Z",
  "dateModified": "2026-06-03T14:32:00Z",
  "image": ["https://gasnow.tools/images/social-card.png"],
  "author": { "@type": "Organization", "name": "GasNow" },
  "publisher": {
    "@type": "Organization",
    "name": "GasNow",
    "logo": { "@type": "ImageObject", "url": "https://gasnow.tools/images/gasnow-icon.png" }
  },
  "mainEntityOfPage": { "@type": "WebPage", "@id": "https://gasnow.tools/news/<slug>" },
  "isBasedOn": "<original source URL>",
  "about": ["Ethereum gas", "Cryptocurrency", "..."]
}
</script>
```

## Google News sitemap

Separate file `news-sitemap.xml` referenced from `robots.txt`. Format:

```xml
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9"
        xmlns:news="http://www.google.com/schemas/sitemap-news/0.9">
  <url>
    <loc>https://gasnow.tools/news/coindesk-blackrock-etf-files</loc>
    <news:news>
      <news:publication>
        <news:name>GasNow</news:name>
        <news:language>en</news:language>
      </news:publication>
      <news:publication_date>2026-06-03T14:32:00Z</news:publication_date>
      <news:title>BlackRock files spot ETH ETF</news:title>
    </news:news>
  </url>
</urlset>
```

Google News sitemaps only accept articles published in the last 48h. Older entries should be moved to the regular sitemap or dropped.

## IndexNow integration

Already shipped (this PR). The cron worker can call:

```bash
npm run indexnow -- https://gasnow.tools/news/<slug-1> https://gasnow.tools/news/<slug-2>
```

…or POST directly to `https://api.indexnow.org/IndexNow` with up to 10 000 URLs per call.

## Open decisions before implementation

- [ ] Are we OK with LLM-generated editorial framing? If yes, which model + cost budget?
- [ ] Where does dedup state live — Supabase table or volume-mounted JSON?
- [ ] How aggressively do we prune (keep last 7 days? 30? forever)?
- [ ] Do we accept the legal risk of summarizing third-party news headlines? (Generally fair use, but worth confirming.)
- [ ] Do we want author byline = "GasNow" or "GasNow Editorial Team" or the original source attributed clearly?

## Effort estimate

- Editorial template + page generator: ~1 day.
- Cron + dedup manifest: ~0.5 day.
- News sitemap + IndexNow wiring: ~0.5 day (most plumbing already exists).
- LLM editorial layer (optional): ~1 day extra.

**Total: 2–3 days of focused work for option A with rule-based framing.**

If we don't have the appetite for the editorial layer, **don't ship this**.
